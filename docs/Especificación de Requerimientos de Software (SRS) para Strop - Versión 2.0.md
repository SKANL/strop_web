# **Especificación de Requerimientos de Software (SRS) para Strop \- Versión 2.0**

## **1.0 Introducción**

### **1.1 Propósito**

El propósito de este documento es especificar de manera completa, precisa y verificable los requerimientos funcionales y no funcionales para el desarrollo de la aplicación de gestión de comunicación e incidencias en obras de construcción, denominada **Strop**. Este documento de Especificación de Requerimientos de Software (SRS, por sus siglas en inglés) sirve como la base formal para las fases subsecuentes de diseño, desarrollo y pruebas del software dentro del Ciclo de Vida del Desarrollo de Software (SDLC). Funciona como un acuerdo formal entre los stakeholders principales (dueños de la constructora, directivos) y el equipo de desarrollo, asegurando que el producto final cumpla con todas las necesidades y expectativas definidas.

### **1.2 Ámbito del Producto**

Strop es una plataforma estratégica diseñada para centralizar y transformar la gestión de la comunicación y el registro de incidencias en proyectos de construcción. El sistema se enfoca en eliminar la comunicación fragmentada y verbal, que carece de validez legal y es ineficiente, para entregar una visión clara, en tiempo real y accionable del estado de cada obra a la dirección.

El alcance del producto abarca tres pilares fundamentales:

- **Visibilidad Instantánea:** Proveer un dashboard ejecutivo que permita a la alta dirección conocer al momento los avances, contratiempos o desviaciones críticas en las obras, sin necesidad de estar físicamente presente en cada una.
- **Comunicación Centralizada:** Establecer un canal de comunicación formal y directo entre el personal de campo y la oficina, permitiendo documentar cada incidencia (solicitudes, imprevistos, reportes de avance) con evidencia fotográfica, asignación de responsables y seguimiento de su estatus en una única plataforma.
- **Registro Inalterable:** Generar un historial cronológico completo y a prueba de manipulaciones de cada incidencia y decisión tomada, emulando la validez legal de una bitácora de obra. Este registro servirá para el deslinde de responsabilidades y el análisis post-proyecto.

#### **Funcionalidad Excluida**

Esta versión inicial del sistema, Strop 2.0, se enfocará exclusivamente en la gestión de proyectos, la comunicación y el control de incidencias. Las siguientes funcionalidades quedan explícitamente fuera del alcance de este documento:

- Gestión de nóminas o control de asistencia de trabajadores.
- Cálculos financieros complejos de estimaciones, presupuestos o ajustes de costos.
- Módulos de contabilidad o inventarios avanzados.

El sistema podrá recibir información financiera (como la explosión de insumos) desde sistemas externos como Opus o Neodata, pero no realizará cálculos sobre ella.

## **2.0 Descripción General**

### **2.1 Perspectiva del Producto**

Strop se concibe como una plataforma independiente y especializada, enfocada en resolver el problema central de la comunicación ineficiente y la falta de control en tiempo real entre la obra y la oficina central. Aunque es un sistema autónomo, está diseñado para interactuar con el ecosistema de software existente en la constructora. Específicamente, deberá poder recibir datos de sistemas de costos y planeación (ej. Opus, Neodata, Project, CIFA, WF5) mediante la importación de archivos en formato estándar (.xlsx, .csv). Esta interacción permitirá cargar la **Ruta Crítica** simplificada y la **explosión de insumos** inicial de cada proyecto, que servirán como la "línea base" inalterable contra la cual se medirá el avance y se controlarán las solicitudes de material.

### **2.2 Funciones del Producto**

Las principales funcionalidades de Strop se agrupan en los siguientes módulos:

- **Módulo de Gestión de Proyectos:** Permite a los administradores crear nuevos proyectos, asignar personal con roles específicos (Superintendente, Residente, Cabo) y cargar el programa de obra (Ruta Crítica) que define la línea base del proyecto.
- **Módulo de Comunicación y Gestión de Incidencias:** Es el núcleo del sistema. Facilita al personal de campo el registro de incidencias (reportes de avance, problemas, solicitudes de material) con evidencia fotográfica georreferenciada y con marca de tiempo, garantizando un canal de comunicación formal y rastreable.
- **Módulo de Dashboard Ejecutivo y Reportería:** Proporciona a la alta dirección (Dueño/Administrador) un panel de control visual que consolida los indicadores clave de rendimiento (KPIs) de todos los proyectos, permitiendo una toma de decisiones rápida e informada sobre desviaciones o problemas críticos.
- **Módulo de Bitácora Digital Inalterable:** Asegura la integridad de toda la información registrada. Cada evento, comentario y cambio de estado queda guardado de forma cronológica e inmutable, creando un historial auditable que sirve como registro formal para el deslinde de responsabilidades.

### **2.4 Restricciones Generales**

- **Soporte Multiplataforma:** El sistema debe funcionar en una plataforma Web para los roles estratégicos y administrativos, y a través de una aplicación móvil (Android/iOS) para el personal de campo.
- **Integridad de la Información:** Está estrictamente prohibido que cualquier usuario, incluido el Administrador, pueda borrar o alterar los registros originales de una incidencia una vez asentada. Solo se permitirán adiciones como notas de seguimiento.
- **Registro Cronológico:** Todas las funcionalidades deben seguir el principio de un registro cronológico, claro y con responsabilidades asignadas, emulando los requerimientos legales de una bitácora de obra.
- **Dependencia de la Ruta Crítica:** El correcto funcionamiento del seguimiento de avances y control de insumos depende de una correcta y completa importación inicial de la Ruta Crítica del proyecto.
- **Operación sin Conexión:** La aplicación móvil debe ser capaz de operar en modo offline, permitiendo al personal de campo registrar incidencias y tomar evidencias sin conexión a internet. Los datos se almacenarán localmente y se sincronizarán de forma automática cuando se recupere la señal.

### **2.5 Suposiciones y Dependencias**

- **Formato de Archivos:** Se asume que la constructora proveerá los archivos del programa de obra (Ruta Crítica y explosión de insumos) en un formato simplificado y estandarizado, compatible con la importación (.xlsx o .csv).
- **Disponibilidad de Hardware:** Se asume que el personal de campo (Residentes, Cabos) contará con dispositivos móviles (smartphones) compatibles con la aplicación (versiones recientes de Android/iOS) y con capacidad de GPS.
